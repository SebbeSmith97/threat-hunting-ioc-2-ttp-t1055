## 1. Hypothesis statement

### Strategic hypothesis

> If an attacker already has access to a system and wants to hide their malicious code inside
> a legitimate process, they will need to plant that code from the outside. That act of
> planting leaves a trace.

This is the big-picture question the hunt is asking. It does not depend on knowing which tool
the attacker used or which process they targeted. It just asks: did something write code into
another process and make it run?

---

### Operational hypothesis

> **If** classical DLL injection is performed, **then** within a short window on the same
> device the same initiating process should exhibit:
> - **(a)** a remote memory allocation into another process (`NtAllocateVirtualMemoryRemoteApiCall`
>   in `DeviceEvents`), followed by
> - **(b)** a remote thread creation targeting that same process
>   (`CreateRemoteThreadApiCall` in `DeviceEvents`), followed by
> - **(c)** a DLL load appearing in `DeviceImageLoadEvents` for the target process.
>
> This behavior is durable because the API sequence is the technique's definition.
> An adversary cannot evade this hunt without abandoning classical DLL injection entirely.

---

### Telemetry

| Table | ActionType | Stage in the injection chain |
|---|---|---|
| `DeviceEvents` | `NtAllocateVirtualMemoryRemoteApiCall` | Memory allocation — attacker allocates space in the target process |
| `DeviceEvents` | `CreateRemoteThreadApiCall` | Thread creation — attacker starts execution in the target process |
| `DeviceImageLoadEvents` | (all rows for target process) | Module load — injected DLL appears in the target's address space |

**Note on telemetry availability:** `NtAllocateVirtualMemoryRemoteApiCall` is generated by
the `Microsoft-Windows-ThreatIntelligence` ETW provider, which requires specific sensor
configuration in Defender for Endpoint. Before relying on Step 1, verify that this
ActionType returns results in your tenant by running:

```kql
DeviceEvents
| where Timestamp > ago(30d)
| where ActionType == "NtAllocateVirtualMemoryRemoteApiCall"
| summarize count() by DeviceName
```

If it returns no results, the hunt can still run on Steps 2 and 3 alone
(`CreateRemoteThreadApiCall` plus `DeviceImageLoadEvents`), though with slightly less
precision on the allocation step.

**Minimum retention required:** 7 days for the hunt window. 30 days if baselining is run
first (recommended).

## 3. KQL query

```kql
// Hunt T1055.001-H01 — Classical DLL injection
// Hypothesis: remote memory allocation + remote thread creation + image load
// in the target process, all within a short window from the same initiator
// Author: Sebastian Smith  |  Version: 1.3  |  Last tested: 2026-05-17

// Step 1 — remote memory allocation into target process (VirtualAllocEx)
let remote_alloc =
    DeviceEvents
    | where Timestamp > ago(7d)
    | where ActionType == "NtAllocateVirtualMemoryRemoteApiCall"
    | where InitiatingProcessId != ProcessId
    | project AllocTime = Timestamp,
              DeviceId,
              DeviceName,
              WriterPid = InitiatingProcessId,
              WriterFile = InitiatingProcessFileName,
              WriterCommandLine = InitiatingProcessCommandLine,
              TargetPid = ProcessId,
              TargetFile = FileName;
// Step 2 — correlate with remote thread creation in the same target (CreateRemoteThread)
let with_remote_threads =
    remote_alloc
    | join kind=inner (
        DeviceEvents
        | where Timestamp > ago(7d)
        | where ActionType == "CreateRemoteThreadApiCall"
        | project ThreadTime = Timestamp,
                  DeviceId,
                  WriterPid = InitiatingProcessId,
                  TargetPid = ProcessId
    ) on DeviceId, WriterPid, TargetPid
    | where datetime_diff('second', ThreadTime, AllocTime) between (0 .. 60);  // allocation then thread, within 60s
// Step 3 — correlate with subsequent image load in the target (LoadLibrary effect)
let full_chain =
    with_remote_threads
    | join kind=inner (
        DeviceImageLoadEvents
        | where Timestamp > ago(7d)
        | project LoadTime = Timestamp,
                  DeviceId,
                  TargetPid = InitiatingProcessId,
                  LoadedFile = FileName,
                  FolderPath
    ) on DeviceId, TargetPid
    | where datetime_diff('second', LoadTime, ThreadTime) between (0 .. 60);  // load follows thread, within 60s
// Step 4 — prevalence filter and aggregation per writer-target-DLL combination
full_chain
| summarize
    Devices   = dcount(DeviceId),
    Events    = count(),
    FirstSeen = min(AllocTime),
    LastSeen  = max(LoadTime),
    WriterCLI = take_any(WriterCommandLine)
    by WriterFile, TargetFile, LoadedFile, FolderPath
| where Devices <= 5                                  // tune this threshold per tenant baseline
| order by Devices asc, Events desc
```

**How to read this query:**

- Step 1 filters for events where one process allocated memory inside another process.
  This is the first step an attacker needs to do before writing their DLL path into the target.
- The first join links those allocation events to thread creation events that happened within
  60 seconds in the same process pair. An allocation with no follow-up thread is likely benign.
- The second join adds the image load event — the DLL actually appearing in the target.
  All three events must occur in sequence for a row to appear in the output.
- `dcount(DeviceId)` counts how many different machines show this writer-target-DLL
  combination. Low counts (5 or fewer) suggest something rare and worth investigating.
  High counts suggest something widely deployed — likely benign, but investigate separately.
- Results are ordered rarest first so the most suspicious combinations surface at the top.

## 4. Expected results and tuning

### 4.1 Expected legitimate causes

The three-step chain has a small but real set of legitimate causes:

- **Endpoint security products** — some AV and EDR sensors inject their own DLLs into
  monitored processes for hooking purposes. The writer process will be a well-known security
  vendor binary such as `MsMpEng.exe`.
- **Debuggers** — Visual Studio, WinDbg and similar tools attach to processes and load
  debugging DLLs. Typically only seen on developer workstations during business hours.
- **IT management tooling** — some remote administration or monitoring agents use injection
  for process instrumentation. These will show consistent `WriterFile` values across the
  fleet and a high `Devices` count.
- **Software that loads plugins into host processes** — some legitimate applications use
  this pattern for plugin loading.

### 4.2 Suggested triage refinements

- Pivot towards the table DeviceFileCertificateInfo to investigate suspicious files.
  If `IsSigned` is true and `Signer` is a recognized software vendor and `Devices` is high,
  check whether this is known-good tooling before escalating.
- If `WriterFile` is a security product from your environment's allow-list, deprioritize.
  Document the allow-list per tenant rather than adding suppression to the base query.
- If the activity occurs on a developer workstation during business hours and `WriterFile`
  is a known IDE, treat as lower priority.
- If `IsSigned` is false (unsigned loaded DLL), treat as higher priority regardless of other
  factors.
- If `TargetFile` is a high-value process such as `lsass.exe`, `svchost.exe`, `explorer.exe`
  or a browser process, treat as higher priority.
- If `WriterCLI` contains encoded strings, unusual paths or scripting engine invocations,
  treat as high priority.

### 4.3 What a true positive looks like

A malicious result typically shows an unusual `WriterFile` (a scripting engine, a
recently-dropped binary, or a process running from a temp or user-writable path) allocating
into a stable system process like `explorer.exe` or `svchost.exe` at an unexpected time.
The loaded `FolderPath` will often be a temp directory, an AppData path or somewhere not
associated with the software implied by the DLL name. The `Signer` field will be empty or
will not match the expected publisher. Follow-on network activity from the target process
shortly after the load is a strong confirming signal.

### 4.4 Recommended pivots

- **Pivot 1 — Loaded DLL details:**
  ```kql
  DeviceImageLoadEvents
  | where DeviceName == "{DeviceName from result}"
  | where InitiatingProcessId == {TargetPid from result}
  | where Timestamp between (datetime({LoadTime}) .. datetime_add('minute', 5, datetime({LoadTime})))
  | project Timestamp, FileName, FolderPath, SHA256, InitiatingProcessFileName
  ```
  Look for unsigned DLLs, unusual paths or DLL names that do not match their expected signer gathered from DeviceFileCertificateInfo.

- **Pivot 2 — Outbound network from the target process:**
  ```kql
  DeviceNetworkEvents
  | where DeviceName == "{DeviceName from result}"
  | where InitiatingProcessId == {TargetPid from result}
  | where Timestamp between (datetime({LoadTime}) .. datetime_add('minute', 30, datetime({LoadTime})))
  | project Timestamp, RemoteIP, RemoteUrl, RemotePort, InitiatingProcessFileName
  ```
  New outbound connections from the injected process, especially to low-prevalence IPs or
  on unusual ports, are a strong indicator of C2 activity.

- **Pivot 3 — Child processes of the target:**
  ```kql
  DeviceProcessEvents
  | where DeviceName == "{DeviceName from result}"
  | where InitiatingProcessId == {TargetPid from result}
  | where Timestamp between (datetime({LoadTime}) .. datetime_add('minute', 30, datetime({LoadTime})))
  | project Timestamp, FileName, ProcessCommandLine, AccountName
  ```
  Shells, scripting engines or reconnaissance tools spawned by the injected process are
  high-confidence true-positive indicators.

## 5. Pyramid of Pain positioning

**Declared level: TTPs**

An adversary performing classical DLL injection cannot evade this hunt without abandoning
the technique entirely. The three steps — allocate memory in a remote process, create a
thread in that process, observe a module load in that process — are not optional
implementation details. They are what the technique is. The specific DLL name can change,
the target process can change, the tooling can be swapped between campaigns. None of that
matters because none of it appears as a filter in the query. The query filters on the API
call sequence, not on any artifact the adversary can rotate cheaply. That is the operational
meaning of TTP-level detection on the Pyramid of Pain.
